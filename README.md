# Digital-Twin-Project
This project aim to automate the process of generating digital twins.


Method Used: Photogrammetry 

Our group decided to use photogrammetry as the method to generate a 3D mesh. We mainly used Meshroom and Blender. 
Meshroom was used to generate the 3D mesh while blender was used to convert the .obj file that was generated by Meshroom to convert into .glb as part of our requirement.
# How does Meshroom produces the 3D digital twin?

First, multiple images of the physical object are captured from different angles. These images must overlap so the same parts of the object appear in several photos.
Meshroom then looks through each image and detects unique visual points, such as corners, edges, patterns, and texture details.
It then compares the images and matches the same visual points across different photos. This allows Meshroom to know which parts of different images belong to the same real-world location on the object.
Using those matched points, Meshroom calculates where each camera was located and how it was oriented when each photo was taken. This step is part of Structure from Motion.
After estimating the camera positions, Meshroom reconstructs an initial 3D structure of the object using points in space. This is called a sparse point cloud.
Meshroom then adds many more points to better describe the object’s surface. This creates a denser and more detailed representation of the real object.
The dense point cloud is converted into a 3D surface made of connected triangles. This surface is called a mesh, and it defines the shape of the digital twin.
Finally, Meshroom projects the original image colors onto the mesh surface. This gives the 3D model a realistic appearance that looks similar to the original object.

# Usage of Blender 5.0

Blender is used to export the model as a .glb file, which gives a nicer output and is more suitable for viewing, sharing, and downstream digital twin applications.

# Description of the setup
The system consists of three Raspberry Pi 4 units (A, B, and C), each equipped with a Raspberry Pi Camera Module 3 connected via FCC (flat flex cable). All three Pis are connected to a network switch via Ethernet cables, which links them to the Host PC. The Host PC also connects wirelessly to a router for internet access.
RPi 4 (B) serves as the master node. In addition to its camera, it is connected to a microstep driver via GPIO jumper wires (DIR and PUL pins). The microstep driver is powered by a 12V 6A DC power supply (230V AC input) and drives a stepper motor configured at 19,700 steps per revolution. The stepper motor rotates the object being scanned in 15° increments across 24 positions to complete a full 360° revolution.
RPi 4 (A) and RPi 4 (C) act as slave nodes — they only handle camera operations and respond to commands from the master.
Software Communication
All three Pis communicate with the Host PC through MQTT, with the broker running on the Host PC (192.168.137.1). The master coordinates the capture sequence: at each rotation step, it signals all Pis to capture simultaneously, waits for confirmation from each, then advances to the next position. Each Pi also runs a Flask web server on port 7123 for live MJPEG video streaming, allowing the host to preview camera feeds via a browser.
